Victor Freundt> Things to consider when programming tiny45, tiny44, mega88, and mega 644. In case you use a XTAL (20 MHz), do not forget to initialize.

avrdude in-circuit programmer
tiny45 load hex file, bsd cable: avrdude -p t45 -c bsd -U flash:w:file.hex
tiny45 load hex file, dasa cable: avrdude -p t45 -P /dev/ttyUSB0 -c dasa -U flash:w:file.hex
tiny45 load hex file, dragon: avrdude -p t45 -P usb -c dragon_isp -U flash:w:file.hex
tiny44 load hex file: avrdude -p t44 -c bsd -U flash:w:file.hex
tiny44 use 20 MHz xtal: avrdude -p t44 -c bsd -U lfuse:w:0×7E:m
mega88 load hex file: avrdude -p m88 -c bsd -U flash:w:file.hex
mega88 use 20 MHz xtal: avrdude -p m88 -c bsd -U lfuse:w:0×76:m
mega644 load hex file: avrdude -p m644 -c bsd -U flash:w:file.hex
mega644 use 20 MHz xtal: avrdude -p m644 -c bsd -U lfuse:w:0×76:m

Check http://fab.cba.mit.edu/about/fab/index.html for more info

Shawn Wallace : Hi Tomas: To run the Thtp bridge, try the settings:

python thtp.py /dev/ttyS0 115200 1234
 (/dev/ttyS0 should be whatever port your serial cable is plugged into)

You’ll find two speeds in the serial bridge assembly code, but the second one is commented out with a semicolon. It is programmed to run at 115200 baud. With the python script running, turn on your leaf nodes. You should get a solid LED if it doesn’t have an I0 address yet. If you don’t get a solid LED, reflash it with the leaf node assembly code. When you press the button, the leaf should be assigned a random address, which should pop up in your python window. Use this address to talk to the node from a web browser by typing this URL in the browser:

http://localhost:1234/[i0 address]:80

Kerry> A. I don’t think the computer needs a fixed IP address. I believe it uses the one it has already.
The serial bridge definitely does not have one. It simply receives a SLIP packet on one port (serial or i0) and sends it out the other one. Serial speed is not a big issue, so long as the PC serial port and the bridge serial port are both set to the same speed! I think you might see the LED on the bridge blink if thtpd.py is running and you try to reach any i0 address using this URL: http://localhost:1234/[any IPv4 address]:80

Of course, you will not get a response unless you’re using the *correct* address for a leaf node. But this is a good test to see if thtp is talking to the serial bridge.
The low fuse on the tiny44 needs to be set to 0×7E before the serial and i0 ports will work correctly.

Kerry: Here’s how I think it’s supposed to work:

1. LED is initially on to indicate no IP address
2. first button press selects random IP address
3. next button press sends set_destination packet (equivalent to “I’m a button”). All other leaf nodes should blink in response.
 4. Pressing a button on any of the other leaf nodes should “bind” it’s LED to the button pressed in 3.
5. At that point, pressing the button should toggle the lamp on and off.

You might want to implement a code change that, when you hold down the button for an extended period of time (say, one second) it reinitializes the leaf node. That will save you from having to reflash in order to reset it.

It sounds like the fourth node you built may have a problem. Try swapping it with a node that you know works and keep everything else constant - same hub, same cable, etc. The simplest way to debug a situation like this is to try and isolate the faulty component.

2. Seven principles

TheScientific American article Krikorian co-authored (cited in “References”) describesseven principles that define Internet-0.

    1. Each Internet-0 device uses theInternet Protocol.
    2. “Implementing thecommunications protocols jointly rather than separately” simplifiessoftware.
    3. Two Internet-0 devices “do not require the existence of a third one in order to operate”.
    4. Each Internet-0 device “is responsible for keeping track of its own identity”.
    5. “The use of big bits allows the data that make up a packet to be represented in the same way no mater what physical medium conveys them.”
    6. Internet-0 uses open standards.
